Declutr — Zero-Trust Digital Life Vault

Software Architecture Document · Principal Engineering Reference

Version 1.0 Classification: Internal Zero-Knowledge Architecture E2E Encrypted

1. System Overview

Declutr is a Zero-Trust Digital Life Vault — a personal archival system that allows users to store, organize, search, preserve, and manage every artifact of their digital life. The system is engineered on a zero-knowledge architecture: all encryption and decryption occurs exclusively on the client device. No Declutr employee, cloud provider, AI subsystem, or system administrator can access plaintext user data at any point.

Core Design Principles

Zero-Knowledge
The server holds only opaque ciphertext. Encryption keys are derived client-side from user credentials using PBKDF2/Argon2id and never transmitted. Server-side operations are cryptographically blind.
Zero-Trust
Every component, service, and infrastructure operator is treated as untrusted. Mutual TLS enforces service-to-service authentication. Access tokens are short-lived and scope-constrained. No implicit trust by network position.
Client-Sovereign Encryption
The client owns all key material. Files are encrypted with AES-256-GCM before leaving the device. A hierarchical key architecture (Master Key → Vault Key → File Key) allows key rotation without re-encrypting every asset.
Privacy-Preserving AI
AI organization and search features operate on the client using WebAssembly-compiled models. For cloud-assisted AI, only encrypted embeddings or blind-compute techniques are used — the model never sees plaintext.
Immutable Auditability
Every server-side action is recorded in a tamper-evident append-only log. Users receive a verifiable audit trail of every access, modification, or deletion event.

Target Users

Privacy-conscious individuals, professionals with sensitive personal archives, families managing multi-generational records, and anyone who needs a structured, permanent home for documents, photos, creative work, health records, and personal correspondence.

Non-Goals

Declutr is not a real-time collaboration platform, not a public sharing service, and not a social network. It is a private, personal archive first. Team vaults and selective sharing are future-phase features, not MVP scope.

4. Authentication Architecture

SRP-6aPasskey / WebAuthnZero-Knowledge Auth

Declutr uses Secure Remote Password (SRP-6a) as its primary authentication protocol. SRP proves knowledge of a password without transmitting it — the server never sees the password or any value from which it can be derived. This eliminates credential theft at the server level.

Registration Flow

1
Client derives salt + verifier
User enters password. Client generates random salt. Derives private key x = H(salt, email, password). Computes verifier v = g^x mod N using SRP group params (4096-bit). Master encryption key derived in parallel via Argon2id(password, salt, params).
2
Send (email, salt, verifier) to server
Server stores verifier v — it is computationally infeasible to recover the password from v. Server never sees the password. The master encryption key stays on the client only.
3
Derive and encrypt Master Vault Key
Client generates a random 256-bit Master Vault Key (MVK). Encrypts MVK with the master encryption key → produces Encrypted Master Vault Key (eMVK). eMVK is uploaded to server and stored — server cannot decrypt it without user's password.
4
Issue account + session tokens
Server issues a short-lived JWT (15 min) + refresh token (30 days, httpOnly, secure, SameSite=Strict). Session bound to device fingerprint and IP subnet.

Login Flow (SRP-6a Exchange)

1
Client → Server: (email, A = g^a mod N)
Client generates ephemeral private a, computes A. Sends to server.
2
Server → Client: (salt, B = kv + g^b mod N)
Server retrieves stored salt + verifier, generates ephemeral b, returns B.
3
Both sides derive shared session key S
Neither side can compute S without the other. Client derives u = H(A, B), then S = (B - kg^x)^(a + ux) mod N. Server derives S = (Av^u)^b mod N.
4
Mutual proof exchange (M1, M2)
Client sends M1 = H(A, B, S). Server verifies M1, replies M2 = H(A, M1, S). Client verifies M2. Both are authenticated. JWT issued. Client decrypts eMVK using locally derived master key.

Multi-Factor Authentication

MethodStandardNotes
PasskeysWebAuthn / FIDO2Preferred. Phishing-resistant. Platform authenticator or hardware key. Replaces password entirely.

Access tokens: JWT (RS256), 15-minute TTL, signed by Auth Service private key. Refresh tokens: opaque random 256-bit token, stored server-side hashed (Argon2id), 30-day TTL. Token rotation on every refresh. Refresh token family invalidation on theft detection.

Key Rotation

Key rotation does not require re-encrypting file content. Only the wrapping keys change:

1
Derive new MEK
Client derives new MEK from new password + new salt (Argon2id).
2
Re-wrap MVK
Decrypt current eMVK → re-encrypt MVK with new MEK → upload new eMVK. Atomic server operation.
3
Vault keys unchanged
eVK (vault keys) remain valid; they are still wrapped by MVK which hasn't changed. File keys unchanged entirely. O(1) operation regardless of vault size.

Metadata Encryption

File names, tags, notes, and folder paths are encrypted with the Vault Key before storage. Server-side search uses blind indexes: BLAKE3-HMAC(VK, normalize(term)) truncated to 32 bytes. Collision probability is negligible; server learns no semantic information about terms.

Content Hash

Client computes BLAKE3(plaintext) before encryption. Hash encrypted with VK and stored server-side. Enables client-side deduplication detection without revealing file identity to server.

6. Vault Architecture

Multi-VaultVersioningStorage Tiers

Vault Hierarchy

User Account ├── Personal Vault (default) │ ├── /Documents │ │ ├── /Financial │ │ ├── /Legal │ │ └── /Medical │ ├── /Photos │ ├── /Creative │ └── /Archive ├── Work Vault (separate key) └── Family Vault (shared key, future phase) Each Vault: ├── vault_id (UUID) ├── encrypted_vault_key (eVK, wrapped by MVK) ├── encrypted_name (AES-256-GCM) ├── encrypted_description └── sealed_metadata_root (HMAC tree root)

Vault Design Decisions

Independent Keys Per Vault
Each vault has its own 256-bit Vault Key. Compartmentalizes exposure: if a vault key is ever compromised (e.g., shared vault), other vaults remain secure. The Master Vault Key wraps all vault keys.
Immutable Versioning
Every file write creates a new version. Versions are content-addressed by their encrypted blob hash. Server enforces append-only semantics for version history. Clients request explicit deletion of specific versions.
Storage Tiering
Hot tier (S3/R2): recent + frequently accessed files. Warm tier: 90-day inactivity transition (S3-IA). Cold tier: 1-year inactivity → Glacier. Archival tier: user-designated "preserve forever" items. Tier classification stored as encrypted metadata.

Folder Metadata Encryption

Folder structure is stored as an encrypted tree. Each folder record contains an encrypted name, encrypted path, and encrypted child manifest. Server sees only UUIDs and encrypted blobs. A client-side recursive walk constructs the displayed hierarchy by decrypting nodes in order.

Deduplication

Client-side convergent encryption for identical files: BLAKE3(plaintext) determines whether a file already exists in the user's vault. If so, only a new eFK (encrypted with the current vault key) is created pointing to the existing blob. Server performs deduplication only across the same user's vault (not cross-user, which would leak information).

7. File Upload Flow

ChunkedResumableClient-Encrypted
1
Client: Generate file key + metadata
Generate random 256-bit File Key (FK). Compute BLAKE3(plaintext). Encrypt metadata (name, size, MIME, hash) with Vault Key. Encrypt FK with Vault Key → produce eFK.
2
Client: Chunk + encrypt
Split file into 4MB chunks. For each chunk: generate 12-byte random IV; encrypt with AES-256-GCM(FK, IV, AAD={file_id, chunk_index, vault_id}); prepend IV + auth tag. All encryption in a dedicated Web Worker thread.
3
Server: Initiate multipart upload
POST /files/initiate → server allocates file_id, creates multipart upload on S3, returns pre-signed URLs for each chunk. Validates JWT scope and quota.
4
Client: Upload chunks in parallel
Upload encrypted chunks directly to S3 via pre-signed URLs (bypasses server for blob transfer). Up to 6 parallel chunk uploads. Exponential backoff retry per chunk.
5
Server: Commit + store metadata
Client sends POST /files/commit with: part ETags, encrypted metadata blob, eFK, blind indexes. Server completes S3 multipart, atomically writes metadata to PostgreSQL. Returns file_id + version_id.
6
Server: Async AI processing request
File commit triggers async job queued for AI processing. Worker sends encrypted file to client-side processor or invokes trusted execution environment for server-side inference (future phase).
7
Client: Write to local cache
Decrypt and store file metadata in local SQLCipher/IndexedDB cache. Update local search index. Display file in vault UI.

Resumable Upload Protocol

Resumable state is stored client-side (IndexedDB): file_id, upload_id, completed chunk set, per-chunk ETags. On resume, client requests /files/:id/status from server to reconcile completed parts. Only missing chunks are re-uploaded. Server-side: uncommitted multipart uploads expire after 7 days.

8. Metadata Architecture

Sealed MetadataBlind Index

Metadata is separated into two layers: sealed metadata (opaque to server, decrypted only by client) and structural metadata (minimally visible to server for routing and storage management).

Sealed Metadata (Server-Opaque)

Encrypted with Vault Key (AES-256-GCM). Contains: file name, original path, MIME type, file size (plaintext size), creation/modification timestamps, user tags, notes, AI-generated labels, GPS data, device info, custom fields. Stored as a single encrypted JSON blob in PostgreSQL.

Structural Metadata (Minimally Visible)

Visible to server for operations: file_id (UUID), vault_id, owner_user_id, version_id, blob_size (ciphertext size in bytes), storage_tier, upload_timestamp (Unix ms), content_hash_blind (BLAKE3-HMAC for deduplication), shard_key (for database partitioning). No content-revealing information.

Blind Index Construction

// For each searchable term: blind_index = BLAKE3_HMAC( key = vault_search_key, // derived from VK input = normalize(term) // lowercase, unicode-normalized ).slice(0, 32) // 32 bytes = 256-bit token // Stored as: { file_id, blind_index[] } // Search: recompute blind_index for query → lookup in DB // Server learns: index collision rate only (not terms)

EXIF / Embedded Metadata

On upload, client strips all EXIF data from images (using a WASM-based EXIF parser) before encryption if the user has enabled "metadata stripping." Stripped EXIF data is separately encrypted and stored as a sub-record — available for search but not transmitted in any unencrypted header or URL.

9. Search Architecture

Encrypted SearchHybrid Local+Server

Search is the hardest problem in zero-knowledge systems. Declutr uses a hybrid approach: local full-text search on cached metadata for small vaults, and server-side blind-index lookup for large vaults, combined at query time.

Search Strategy by Vault Size

Vault SizeStrategyPrivacyLatency
< 10,000 filesFull local search (IndexedDB / SQLCipher)Perfect — server never queried<50ms
10k–500k filesHybrid: local cache + server blind-indexServer sees query tokens only<300ms
> 500k filesServer blind-index primary, local refinementServer sees access pattern<800ms

Local Search Stack

FlexSearch.js compiled to WASM, running in a Web Worker. Indexes decrypted metadata in memory (not persisted). Re-built on app load from local SQLCipher cache. Supports full-text, fuzzy match, field-weighted ranking. Completely private — no server involvement.

Server Blind-Index Search

1
Client tokenizes + normalizes query
Tokenize query into terms. Apply same normalization as indexing (lowercase, unicode NFC, stopword removal).
2
Compute blind tokens
For each term: blind_token = BLAKE3_HMAC(vault_search_key, term). These are deterministic pseudorandom values — server cannot reverse to original terms.
3
Server lookup + intersection
Server performs indexed lookup on blind_index table. Returns file_ids matching any/all blind tokens. Server does not know what the tokens represent.
4
Client decrypts + ranks results
Client fetches + decrypts metadata for matched file_ids. Applies local ranking: TF-IDF score, recency boost, file type boost. Renders results.

AI-Enhanced Search (Local)

Semantic search uses client-side MobileCLIP embeddings. On upload, client generates a 512-dim embedding vector from file content → encrypts vector → stores encrypted embedding. At search time, client decrypts local embeddings and computes cosine similarity entirely in-memory. No semantic vectors ever reach the server.

Cloud AI (Optional / Future)

For users who opt in to cloud AI on large files: file is re-encrypted inside an Intel TDX / AWS Nitro Enclave. AI model runs inside the enclave. Plaintext never leaves the enclave. Enclave attestation verified by client before sending data.

11. Digital Legacy Architecture

Dead Man's SwitchShamir Secret Sharing

Declutr's Digital Legacy system allows users to designate beneficiaries who receive access to specified vault contents after a confirmed death or incapacity event. The system is designed so Declutr itself cannot unilaterally execute a legacy transfer — beneficiary authorization is cryptographically gated.

Legacy Packet Construction (Client-Side)

1
Select vault(s) + beneficiary
User selects which vaults or folders to include. Designates beneficiaries by email/public key. Defines trigger conditions.
2
Generate legacy key (LK)
Client generates random 256-bit Legacy Key. Encrypts selected vault keys with LK → produces Encrypted Legacy Vault Keys (eLVK). LK is never stored directly.
3
Split LK with Shamir Secret Sharing
Apply (k, n) Shamir: e.g., 2-of-3. Shares distributed to: Declutr escrow server (1 share), designated beneficiary (1 share), trusted third party/attorney (1 share). No single party has enough to reconstruct LK.
4
Upload encrypted legacy packet
eLVK + encrypted beneficiary instructions uploaded to server. Server stores Declutr's SSS share. Packet is opaque to server.

Legacy Trigger Mechanisms

TriggerMechanismVerification Required
InactivityNo login for N days (user-configured: 1–5 years). Grace period with notification.Multi-party confirmation required before release
Legal documentBeneficiary submits certified death certificate via notarized uploadDeclutr legal review + beneficiary SSS share
Manual activationUser provides emergency code to beneficiary (printed recovery kit)Emergency code + beneficiary SSS share

Legacy Execution

When trigger conditions are verified, Declutr releases its SSS share to the beneficiary's authenticated session. The beneficiary must provide their own SSS share. Combined shares reconstruct LK → beneficiary client decrypts eLVK → gains access to designated vault contents. Declutr cannot access contents without the beneficiary's share.

14. Security Model

Defense in DepthZero-Trust

Security Layers

Layer 1: Client Encryption
AES-256-GCM with per-file random keys. All encryption/decryption in WASM isolated context. WebCrypto API with extractable=false for key objects — keys cannot be exfiltrated via JS even with XSS.
Layer 2: Transport Security
TLS 1.3 mandatory. Certificate pinning on native apps. HSTS with preloading. No TLS 1.2 downgrade accepted. Certificate Transparency monitoring.
Layer 3: API Security
JWT RS256 with 15-min expiry. Scope-restricted tokens (upload:write, vault:read, etc.). HMAC request signing for sensitive operations. CORS restricted to origin allowlist only.
Layer 4: Network Segmentation
VPC with private subnets for all data layer components. API Gateway in public subnet, services in private subnet. No direct internet access from database/storage subnets. NAT gateway for outbound only.
Layer 5: Data-at-Rest
AES-256 server-side encryption for all S3 objects (SSE-S3 as baseline + client encryption on top). PostgreSQL tablespace encryption. Redis: TLS + AUTH + no persistence of sensitive cache keys.
Layer 6: Operational Security
Engineers access production via PAM (Privileged Access Management) with session recording. No persistent access to production. All database queries logged. No production data in dev/test environments. Annual penetration testing.

Threat Scenarios (STRIDE)

Spoofing — Credential TheftMitigated
SRP-6a eliminates password transmission. Server compromise yields only verifier (computationally infeasible to invert). Passkey/WebAuthn provides phishing-resistant secondary path.
Tampering — File Content ModificationMitigated
AES-GCM authentication tag detects any ciphertext modification. S3 object checksums detect storage-layer corruption. Version history provides restoration path.
Repudiation — Action DenialMitigated
HMAC-chained audit log is tamper-evident. Any gap or modification in the chain is detectable. Users can export and verify their audit log independently.
Information Disclosure — Server BreachMitigated
Server stores only ciphertext (files) + sealed metadata + blind indexes. A full server compromise reveals: user IDs, upload timestamps, ciphertext sizes, blind tokens (meaningless without VK). No plaintext content disclosed.
Denial of Service — API FloodingPartially Mitigated
Rate limiting (token bucket), Cloudflare DDoS protection, regional autoscaling. Remaining risk: sophisticated low-rate distributed attacks. Mitigation: PoW challenge tokens for unauthenticated endpoints (v2).
Elevation of Privilege — JWT ForgeryMitigated
RS256 asymmetric signing. Private key in HSM (AWS CloudHSM). Public key distributed for verification. Algorithm confusion attacks prevented: alg field validated server-side, none/HS256 rejected.
Weak KDF — Brute Force on eMVKMitigated
Argon2id with memory=64MB, iterations=3, parallelism=4. At this cost, a consumer GPU performs ~8 guesses/second. A 12-char random password has >2^70 entropy — computationally infeasible to brute force.

Database Sharding Strategy

All tables shard on user_id. Cross-shard queries are eliminated by design — all user data lives in the same shard. Blind index lookups are scoped by vault_id (which includes user_id shard key). File metadata is co-located with user record. Shard key is embedded in all IDs using a prefix scheme.

Multi-Region Architecture (v2)

Users select a primary region (EU, US, APAC) for data residency. Vault data stored exclusively in selected region. Metadata database replicated within region only. Auth service federated across regions via edge-cached public keys. Latency-based DNS routing directs users to nearest regional cluster. GDPR/data sovereignty enforced by region isolation.

Storage Cost Optimization

Intelligent tiering policy: files not accessed in 90 days → S3-IA (40% cost reduction). Files not accessed in 1 year → Glacier Instant Retrieval (68% reduction). User-designated archive files → Glacier Deep Archive (90% reduction). Storage analytics (client-computed) help users identify large/duplicate files. Deduplication within user vaults avoids double-charging identical content.

Post-Quantum Cryptography Migration

Timeline: NIST PQC standards (ML-KEM, ML-DSA) finalized in 2024. Declutr migration path:

1
Phase 1 (2025): Hybrid KEM
Add ML-KEM-768 alongside X25519 for key wrapping. Hybrid mode: classical + PQC. No migration required — additive security.
2
Phase 2 (2026): PQC Auth
Migrate JWT signing to ML-DSA-65. Update SRP to incorporate PQC commitment scheme.

Deprecate classical-only key exchange. All new vaults PQC-native. Legacy vault key rotation offered to existing users.
